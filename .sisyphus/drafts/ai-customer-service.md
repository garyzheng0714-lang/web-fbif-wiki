> 讨论日期：2026-03-17
> 基于：`docs/ai-customer-service-research-report.md` 调研报告

## 一、项目背景

### 已有基础设施
- `fbif-wiki`：Next.js 14 项目，已部署在阿里云 ECS
- 飞书深度用户，已有飞书 OAuth 认证、飞书 API 集成
- PostgreSQL、Redis、MinIO 存储栈
- GitHub Actions 自动部署流程

### 新建模块
为 FBIF 活动搭建 AI 客服系统，服务参会者咨询和合作方/赞助商咨询。

### 关键约束
| 约束 | 详情 |
|------|------|
| QA 数据量 | 2000+ 条 |
| 更新频率 | 每天都在变 |
| 维护方式 | Claude 辅助开发和维护，用户做决策和管理 |
| 时间节奏 | 1-2 个月，不急但有明确目标 |
| 生态偏好 | 深度飞书用户，优先飞书生态方案 |

## 二、最终架构决策

### 确认的技术选型

| 层级 | 方案 | 角色 |
|------|------|------|
| 数据层 | 飞书多维表格 | 运营团队直接编辑 QA 内容，零学习成本 |
| 知识库 | Coze 文档型知识库 | 语义检索（RAG），支撑 AI 回答 |
| 同步层 | 自建同步脚本（TypeScript） | 飞书表格 → 格式化 TXT → 上传 Coze 知识库 |
| AI 对话 | Coze Bot + Chatflow 工作流 | 知识库检索 → 大模型生成回答 → 降级判断 |
| 渠道分发 | Coze 原生多渠道 | 官网嵌入、微信公众号、飞书机器人等 |
| 反馈闭环 | Chatflow HTTP 节点 → 自建日志接收 API | 对话日志回流飞书审核表 |
| 部署 | 阿里云 ECS + PM2 | 同步脚本 + 日志接收 API |

### 架构图

```
飞书多维表格（运营编辑 QA 内容）
    ↓ 同步脚本（分类文件 + 交换替换）
Coze 文档型知识库（语义检索）
    ↓
Coze Bot Chatflow（检索 → 生成 → 降级判断 → HTTP 推送日志）
    ↓
多渠道发布（官网 / 微信公众号 / 飞书机器人）
    ↓ HTTP 推送
自建日志接收 API（阿里云 ECS）→ 飞书审核表 → 人工审核 → 回写 QA 主表
```

### 同步策略（针对调研报告的优化）

调研报告原方案是"删旧文档 → 上传新文档"，存在空档风险。优化为：

1. **按分类拆成多个文档**（如"通用问题.txt"、"2026活动信息.txt"、"赞助商FAQ.txt"），每次只替换变更的分类
2. **交换替换而非删后再传**：先上传新文档 → 确认上传成功 → 再删除旧文档，避免知识库空档
3. **定时 + 增量判断**：同步脚本通过飞书 `last_modified_time` 筛选变更记录，只重新生成被修改分类的文件

## 三、已排除的方案及原因

### Dify（自部署 RAG 平台）

| 评估点 | 结论 |
|--------|------|
| 知识库能否直连飞书多维表格 | **不能**，只支持文件上传（PDF/TXT/CSV），同步问题与 Coze 一样 |
| 多渠道发布 | **不支持**，没有微信/飞书一键发布，需自己写对接代码 |
| 运维负担 | Docker Compose 启动 7+ 容器（Web、API、Worker、PostgreSQL、Redis、向量数据库、Nginx） |
| 与 Coze 关系 | 功能高度重叠，用 Dify 等于放弃 Coze 的多渠道优势 |

**结论：Dify 没有解决 Coze 解决不了的问题，反而丢掉 Coze 最大优势（多渠道一键发布），增加运维负担。不适合本场景。**

### pgvector（PostgreSQL 向量扩展）

| 评估点 | 结论 |
|--------|------|
| 安装扩展本身 | 简单，一条 SQL |
| 实际可用所需工作 | Embedding 服务（API 调用 + Key 管理）、文本分段逻辑、向量索引调优、检索排序逻辑、对外暴露检索 API、内容变更时重新向量化 |
| 总工作量 | 等同于自建完整 RAG 管线，≈ 自部署 Dify 的工作量 |

**结论：pgvector 扩展轻量，但围绕它的完整 RAG 管线不轻量。在有 Coze 知识库可用的情况下，自建 RAG 投入产出比不合理。**

### Coze 表格型知识库

**结论：调研报告已确认不支持行级 CRUD，只能上传整个文件替换。已排除。**

### Coze 数据库 + SQL 检索

**结论：支持行级 CRUD 和增量更新，但检索方式是 SQL 精确匹配而非语义检索，面对用户多样化的提问方式，回答质量可能不足。作为后续备选，第一版不采用。**

### 直查飞书多维表格（无同步层）

| 评估点 | 结论 |
|--------|------|
| 架构简洁度 | 极高，完全砍掉同步层 |
| 数据实时性 | 完美，改了立即生效 |
| 检索能力 | 飞书 Search API 是关键词匹配，非语义检索 |

**结论：适合精确信息查询（日期、价格、联系方式），但无法应对开放式问题。作为后续"混合方案"的组成部分保留，第一版暂不引入。**

## 五、后续升级路径

当系统跑起来、收集到真实对话数据后，可根据实际情况考虑：

1. **如果 Coze 知识库检索质量不够好** → 评估在 Chatflow 中加入"直查飞书多维表格"的 HTTP 节点，精确数据走直查、开放问题走知识库
2. **如果需要更精细的检索控制** → 彼时再评估是否自建检索层（届时技术选型可能已有更成熟的方案）
3. **如果 Coze API 调用量不够** → 升级专业版，或评估调用频率优化空间
4. **如果需要微信小程序** → 需要企业资质，优先用微信公众号覆盖

---

## 六、核心原则

> **先跑通再优化。** 不在没有真实数据的情况下做过度架构决策。第一版用最简方案上线验证 AI 回答质量和用户真实需求，再根据数据驱动迭代。
